# DeepSeek Ask CLI Design

Date: 2026-06-18

## Goal

Add a `mycli deepseek ask` command that sends prompts through the user's logged-in DeepSeek web page and prints only the final answer.

The command supports three modes:

- `instant`
- `expert`
- `vision`

The default mode is `instant`. Vision mode supports one or more image attachments. Expert mode must not include DeepSeek's visible thinking text in CLI output.

## User Interface

### Command

```sh
mycli deepseek ask --text "question"
mycli deepseek ask --file ./prompt.md --mode expert
mycli deepseek ask --text "describe these images" --mode vision \
  --attach ./one.png \
  --attach ./two.png
```

### Options

- `--text <prompt>` supplies the prompt directly.
- `--file <path>` reads the prompt from a UTF-8 text file.
- `--text` and `--file` are mutually exclusive. One is required.
- `--mode <mode>` accepts only `instant`, `expert`, or `vision`.
- `--mode` defaults to `instant`.
- `--attach <path>` and `--attachment <path>` may be repeated.
- Repeated attachments retain command-line order.
- `--timeout_ms <milliseconds>` and `--wait <milliseconds>` follow the existing `doubao ask` timeout behavior.
- The shared `--account <name|id>` option continues to select among multiple connected DeepSeek pages if needed.

The command writes the final answer to stdout and appends one trailing newline when necessary. Progress and diagnostic messages remain outside stdout.

## Architecture

The implementation follows the existing browser-bridge architecture:

1. `clis/deepseek/ask.js` validates CLI options and sends an `ask` command to the local daemon.
2. `src/daemon.js` serves local attachments to the connected page bridge using its existing attachment mechanism.
3. `clis/deepseek/userscript.user.js` runs in the logged-in `chat.deepseek.com` page.
4. The userscript creates a new chat, selects the requested mode, uploads attachments, enters the prompt, submits it, and waits for the final answer.
5. The userscript returns only the final answer through WebSocket.
6. The CLI prints the returned answer.

This design deliberately uses the web UI rather than DeepSeek's private API. The observed private completion flow uses `/api/v0/chat/completion` with fields including `model_type`, `prompt`, `thinking_enabled`, and `search_enabled`, but it also depends on the page's login token, proof-of-work challenge, and WAF state. UI driving keeps those mechanisms inside DeepSeek's own client.

## Components

### Command registration

Add:

- `clis/deepseek/index.js`
- `clis/deepseek/ask.js`

The command implementation should reuse the established `doubao ask` conventions for:

- prompt loading
- timeout validation
- repeated attachment parsing
- attachment file validation
- daemon command dispatch
- stdout formatting

DeepSeek-specific validation adds strict mode normalization. Mode matching is case-sensitive and invalid values fail before contacting the daemon.

### DeepSeek userscript

Add `clis/deepseek/userscript.user.js` with:

- `@match https://chat.deepseek.com/*`
- the standard mycli daemon WebSocket connection
- leader locking consistent with other single-page workers
- a status overlay consistent with existing site bridges
- account metadata when a stable non-sensitive account label can be read from the page

The userscript must not read or transmit browser cookies, local storage, access tokens, passwords, or other authentication material. It only controls the already-authenticated page.

### Mode selection

The userscript maps CLI modes to the visible DeepSeek controls:

| CLI mode | DeepSeek control |
|---|---|
| `instant` | `Instant` |
| `expert` | `Expert` |
| `vision` | `Vision` |

Before sending, the userscript verifies that exactly one matching mode control exists, activates it, and verifies the selected state. It must fail with a clear error if the requested control is unavailable.

The implementation should prefer stable semantic attributes such as radio labels and checked state. CSS class names generated by the frontend are fallback-only because they are not stable.

### Conversation isolation

Every `ask` command starts a new DeepSeek conversation before entering content. This prevents previous messages from affecting the result and gives each CLI invocation a clear answer boundary.

The userscript waits until the new-chat state is ready and the message input is editable before proceeding.

### Attachment upload

Attachments use the daemon's existing local attachment URLs.

The userscript:

1. Downloads each attachment through `GM_xmlhttpRequest`.
2. Reconstructs a browser `File` using the supplied name and MIME type.
3. Assigns files to DeepSeek's visible file input in original CLI order.
4. Waits until uploads finish and the files appear attached before submitting.

Vision mode is the primary image workflow and supports multiple images.

Instant and Expert modes may receive attachments because DeepSeek currently supports extracting text from compatible files in those modes. If the active DeepSeek UI disables upload for the requested mode, the userscript returns a mode-specific error instead of silently changing modes.

No automatic mode switching occurs based on attachment type.

### Prompt submission

The userscript fills the visible `Message DeepSeek` input and submits using the page's enabled send control.

Before submission it records an answer baseline containing the current conversation URL and existing assistant-response candidates. This baseline prevents returning stale text from another message.

It verifies that:

- the requested mode remains selected
- all attachments are ready
- the prompt is present
- the send control is enabled

### Answer detection

The answer collector must distinguish these areas:

- user prompt
- Expert thinking/reasoning
- final assistant answer
- search/tool status
- footer text such as `AI-generated, for reference only`

Only the final assistant answer is returned.

For Expert mode, visible content under labels such as `Thinking`, `DeepThink`, reasoning containers, or collapsible thought sections is explicitly excluded. The collector should identify the final response container structurally rather than subtracting text from a combined page dump.

Completion requires both:

1. DeepSeek no longer shows an active stop/generating control for the current response.
2. The final-answer text has remained unchanged for a short quiet period.

The quiet period protects against the UI finishing the network stream before the rendered Markdown has fully flushed. Empty final answers are not treated as success.

The result should preserve meaningful Markdown text where the DOM exposes it, while excluding controls, citations panels, copy buttons, and status labels.

## Data Flow

```text
CLI options
  -> validate prompt, mode, timeout, and attachment paths
  -> daemon command with prompt, mode, and attachment metadata
  -> connected DeepSeek userscript
  -> new conversation
  -> select mode
  -> upload attachments
  -> submit prompt
  -> wait for stable final answer
  -> WebSocket result
  -> stdout
```

The command payload is:

```json
{
  "prompt": "question",
  "mode": "expert",
  "wait_ms": 300000,
  "attachments": [
    {
      "id": "0",
      "name": "one.png",
      "mime": "image/png",
      "size": 12345,
      "url": "/attachment/command-id/0"
    }
  ]
}
```

Local paths are removed by the daemon before dispatch. The browser receives only daemon-served attachment metadata.

## Error Handling

The CLI rejects:

- missing `--text` or `--file`
- simultaneous `--text` and `--file`
- invalid `--mode`
- missing attachment values
- nonexistent or non-file attachment paths
- oversized attachments according to the daemon's existing limit
- invalid timeout values

The userscript returns actionable errors for:

- no connected DeepSeek page
- page not logged in
- mode control missing or not selectable
- new chat not becoming ready
- attachment input missing
- attachment rejected or upload not completing
- prompt input missing
- send control disabled
- DeepSeek CAPTCHA or verification challenge
- response timeout
- generation ending without a final answer
- page structure changing enough that answer extraction is unsafe

CAPTCHAs and account verification are never bypassed. The error directs the user to complete the check in the browser and retry.

## Testing

Add Node tests for pure and CLI-side behavior:

- default mode is `instant`
- `instant`, `expert`, and `vision` are accepted
- other mode values are rejected
- `--text` and `--file` validation
- timeout selection and validation
- repeated `--attach` and `--attachment` preserve order
- attachment metadata is sent with `mode`
- answer output receives exactly one trailing newline

Extract userscript logic that does not require a live DOM into a small testable module where practical. Cover:

- mode normalization and label mapping
- final-answer candidate filtering
- exclusion of thinking/status text
- answer stability state transitions

Run:

```sh
npm test
npm run check
```

Manual verification uses a logged-in DeepSeek page and covers:

1. Instant text prompt.
2. Expert text prompt with visible thinking; stdout contains only the final answer.
3. Vision prompt with one image.
4. Vision prompt with multiple images and preserved ordering.
5. Invalid mode.
6. Timeout behavior.
7. Upload rejection.
8. CAPTCHA or verification interruption.

## Compatibility and Scope

This change does not expose DeepSeek's private API or authentication state. It does not add conversation continuation, history selection, streaming stdout, search toggles, DeepThink toggles independent of mode, or direct API mode.

The first version targets the current DeepSeek web UI observed on June 18, 2026. DOM selectors should be centralized so future UI changes are localized to the userscript.
